I. Theorem + Proof Sketch

(event-historical multiple worldline traversal)

Theorem (Persistence and Selective Extension of Worldline Realizations)

Let 𝓗 be an irreversible event history generated by a sequence of authorial acts.
Let {WLᵢ} be a finite set of representational realizations (documents) derived from 𝓗 under distinct constraint fields {𝓒ᵢ}.

Then:

Each WLᵢ is a worldline realization compatible with 𝓗.

The projection π : WLᵢ → 𝓗 is many-to-one and non-invertible.

Selection of a realization for continuation does not eliminate other realizations.

Future evolution extends exactly one WLᵢ at a time, while others remain admissible but inactive.

Proof Sketch

Irreversibility of 𝓗
Once a document version exists (draft, completed, journal-specific), the act of its creation is irreducible. No future operation can restore a state in which it did not exist. Thus all versions contribute permanently to 𝓗.

Multiplicity without branching
Each document WLᵢ encodes the same argumentative events but embedded under a different constraint field 𝓒ᵢ (audience norms, formal expectations, evaluation criteria). Hence multiplicity lies in embedding, not in history.

Non-invertibility
From any single WLᵢ, the full set {WLⱼ} cannot be reconstructed. The projection π erases constraint-specific information. Thus traversal does not permit counterfactual recovery.

Selective extension
Choosing one version for further work allocates future authorial acts exclusively to that WLᵢ. This is not collapse, but continuation selection. Other WLⱼ persist but acquire no new events.

Therefore, multiple worldline realizations coexist over a single irreversible history, with traversal realized as selective extension under constraints. ∎

III. Why This Is Not Git Branching (Sharp Distinction)

This is important, because reviewers will think “this is just version control.”

Git Branching Ontology (State-First)

Assumes a single privileged present

Branches are temporary divergences

Merge erases distinction

History is editable, squashable, rewriteable

Branches are means to an end

Your Practice (Admissibility-First)

Assumes irreversible accumulation

Versions are persistent realizations

No merge is required or even meaningful

Selection allocates future events, not truth

Versions are ends in themselves

Key distinction:

Git asks “which version is correct?”
Your framework asks “which realization will be extended next?”

This is the difference between state selection and history continuation.
